## 🏛️ Methodologies (The "How")

### 1. The Verb Cipher Protocol
**Source:** `01_Protocol_The_Verb_Cipher.md`
**Concept:** A filter that determines study depth based on the exam objective's command verb.
*   **Level 1 (Knowledge):** "Explain/Compare" -> Flashcards Only.
*   **Level 2 (Application):** "Configure/Use" -> Lab w/ Syntax focus.
*   **Level 3 (Analysis):** "Troubleshoot" -> Break-Fix Scenarios.

### 2. The 3D Coordinate Schema
**Source:** `Brainstorming_Analysis_Strategy.md`
**Concept:** Transforming the linear exam list into a structured database where every topic has three coordinates:
*   **Vertical (Lens):** Which Tree does it feed? ([PHY], [LOG], [PRF])
*   **Horizontal (Tool):** Which instrument inspects it? (Obj 5.5)
*   **Depth (Method):** Which stage of the lifecycle is it? (Obj 5.1)

## 🧠 Theories (The "Why")

### 3. The "Three Trees" of Troubleshooting
**Source:** `Objective_Architecture_Brainstorming.md`
**Concept:** Domain 5 is not a list; it is three distinct "root systems" that all other domains feed into.
*   **[PHY] Physical Reality (5.2):** Hardware, Cabling, Power.
*   **[LOG] Logical Reality (5.3):** Protocols, Addressing, Services.
*   **[PRF] Performance Reality (5.4):** Flow, Capacity, Latency.

### 4. Domain 5.0 as the "Master Key" (Composite Skills)
**Source:** `Objective_Architecture_Brainstorming.md`
**Concept:** Troubleshooting is the "Integration Test" for Domains 1-4.
*   **Strategy:** Use Domain 5 as a Diagnostic Tool. If you fail a troubleshoot, you "fetch" the specific dependency from the earlier domains. This creates a "Just-in-Time" learning path.

### 5. Dynamic Lens Inference
**Source:** `Brainstorming_Analysis_Strategy.md`
**Concept:** Objects are not static. A "Router" changes attributes based on the "Lens" (e.g., Security Lens vs. Hardware Lens).
*   **Application:** We study only the attributes relevant to the current active lens to prevent "Context Drift."

## 🛠️ Tactics (The "What")

### 6. Objective-Centric Extraction
**Source:** `Brainstorming_Analysis_Strategy.md`
**Concept:** Re-assembling linear textbook chapters into functional "Study Packets" that group all info about an entity (e.g., Router) from every chapter into one view.

### 7. Faceted Knowledge Matrices
**Source:** `Brainstorming_Analysis_Strategy.md`
**Concept:** Building comparison tables that query objects through specific lenses.
*   *Example:* Comparing "Switch vs. Router" specifically through the "Security Lens" (Port Security vs. ACLs).

### 8. The "Void" Check
**Source:** `01_Protocol_The_Verb_Cipher.md`
**Concept:** Scanning the textbook for "Level 3" (Troubleshoot) labs. If missing, we must generate them using AI/Packt.

### 9. Contextualized Embeddings
**Source:** `geminiTranscript0_2-1-26.md`
**Concept:** Domains 1-4 are the "Raw Data Set." Domain 5 is the "Vector Embedding" that gives that data direction and context.

### 10. The Appendix Integration Strategy
**Source:** `Appendix_B_Subnets.pdf`, `Appendix_C_Practice_Exam.pdf`
**Concept:** Appendices are not "extra reading"; they are specialized functional tools.
*   **Appendix B (Subnetting):** The "Calculation Protocol" for the [LOG] Tree. Standalone module for mastering IP math (Obj 1.4).
*   **Appendix C (Practice Exam):** The "Validation Instrument." Used to reverse-engineer "Answer Depth" (how deep do they actually ask?) rather than just testing knowledge.
